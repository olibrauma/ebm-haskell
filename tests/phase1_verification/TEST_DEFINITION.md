# Phase 1 テスト戦略と検証パラメータの定義

## 1. 検証アプローチ: ゴールデンマスター・テスト
リファクタリング前後のバイナリが、**全く同一のパスで全く同一の内部状態を出力すること**を確認します。
シミュレーションの完走を待つのではなく、指定したステップ数（例: 100ステップ）における全状態量をダンプし、ビットレベルで比較（または許容誤差内での比較）を行います。

## 2. 検証用パラメータ・グリッド (Test Grid)
以下の変数を組み合わせた全 36 通りのケースで、リファクタリングの正当性を確認します。これらは惑星科学的にありうる境界条件を網羅しています。

| パラメータ | テスト値 | 意味・意図 |
| :--- | :--- | :--- |
| **P_total [bar]** | 0.007, 0.1, 1.0 | 標準火星、中圧、高圧環境 |
| **obl [deg]** | 0.0, 25.19, 45.0, 90.0 | 垂直、標準傾斜、高傾斜、横転惑星 |
| **alpha [deg]** | 0.0, 30.0, 45.0 | 平坦地、標準斜面、急斜面 |

## 3. ダンプ対象の状態量 (State Variables)
各ステップ、または最終ステップにおいて以下の全データをダンプします。
- スカラ: `P_air`, `P_ice`, `P_rego`, `season`, `T_sub`, `loop`
- 配列 (181要素): `T[]`, `M[]`, `T_posi[]`, `M_posi[]`, `T_nega[]`, `M_nega[]`

## 4. 実行手順
1. **Origin保存**: オリジナルの `test.c` にダンプコードを埋め込み、全グリッドパターンを実行して `tests/results/origin/` に保存する。
2. **Refactor実行**: リファクタリング後の `test.c` で同様に実行し、`tests/results/refactored/` に保存する。
3. **比較**: `scripts/verify_results.py` を実行し、全てのファイルが一致することを確認する。

## 5. 判定基準
- **絶対誤差**: すべてのスカラおよび配列要素において $10^{-14}$ 未満であることを合格とする。
- **一貫性**: 構造体の導入がロジックに影響を与えていない場合、結果は浮動小数点の計算順序も変わらないため、実質的に完全一致することが期待される。
