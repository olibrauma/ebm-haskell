# リファクタリングの全貌とフェーズ分け

## 1. リファクタリングの全貌 (Full Scope)

現在の `test.c` は全ての処理が一つのファイルに詰め込まれ、状態管理が個別変数や多数の配列に分散しています。これを、科学的な計算精度を維持したまま、以下の4つの観点で再構築します。

1.  **カプセル化 (Encapsulation)**: `GlobalState`（温度、氷量、気圧）と `PlanetConfig`（軌道要素、物理定数）に整理し、関数の引数を構造体ポインタに統一する。
2.  **モジュール化 (Modularization)**: 物理プロセス（日射、大気放射・拡散、相変化）ごとにファイルを分割し、各モジュールの結合度を下げる。
3.  **定数の外部化 (Externalization)**: コード内に散在するマジックナンバー（太陽定数、惑星質量等）を構造体メンバーとして管理し、初期化関数でまとめて設定する。
4.  **コードの健全化 (Code Health)**: 未使用変数の削除、適切な変数名への変更、一貫したエラーハンドリングの導入。

## 2. フェーズ分け (Phasing)

可能な限り「動かない期間」を短くし、各段階で既存の出力（Golden Master）と比較検証しながら進めます。

### [Phase 1] データ構造の導入と機能の純粋化
- **目標**: 既存のロジックを1ミリも変えず、データ保持形式を配列・個別変数から構造体へ移行する。
- **内容**: 構造体定義の追加と、全関数のシグネチャ変更。この段階ではファイルは `test.c` 1つのままとする。

### [Phase 2] 物理プロセスごとのモジュール分割
- **目標**: ソースコードを論理的な単位で分割し、機能的な見通しを良くする。
- **内容**: `insolation.c`, `atmosphere.c`, `phase_change.c`, `main.c`, `ebm.h` への分割。Makefile の導入。

### [Phase 3] 定数の整理とインターフェースの洗練
- **目標**: ハードコードされた数値を排除し、シミュレーション設定を柔軟に変更可能にする。
- **内容**: 物理定数を構造体初期化へ移行。入出力処理の共通化。

### [Phase 4] 健全化と最適化
- **目標**: 警告の完全排除と、保守性の最終向上。
- **内容**: 未使用変数の削除、型の一貫性（`int` vs `size_t` 等）の確保。
