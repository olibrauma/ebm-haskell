# リファクタリング戦略と実施計画 - EBM-on-Mars

## 1. 目的
本計画は、計算負荷の高い惑星気候シミュレーションプログラム（`test.c`）を、科学的な妥当性を維持しつつ、保守性と拡張性を高めるためにリファクタリングする手順を定めたものです。他の AI エージェントが本計画を参照するだけで、確実かつ等価なテストを伴う作業を遂行できることを目標とします。

## 2. リファクタリングの基本方針
### 2.1 状態と設定のカプセル化
グローバル変数や `main` 関数内の大量のローカル配列をまとめ、構造体を用いて管理します。
- `PlanetParams`: 軌道要素（傾斜角 `obl`）、太陽定数、大気総量などの物理定数。
- `ClimateState`: 温度 `T`、二酸化炭素氷量 `M`、各大気圧成分などの変動する状態量。
- `SimConfig`: 解像度 `reso`、時間ステップ `dt` などのシミュレーション設定。

### 2.2 物理過程のモジュール化
物理プロセスごとにソースファイルを分割し、各関数が明確な責任を持つようにします。
- `insolation.c`: 日射量計算。
- `atmosphere.c`: 拡散、放射、温度分布計算。
- `phase_change.c`: 相変化（氷・レゴリス）計算。
- `simulation.c`: 主ループおよび平衡状態計算（`heikou`）。

### 2.3 パラメータの外部化
関数内でハードコードされている物理定数（`Q=1366.0` など）を `PlanetParams` に移行し、テスト時に柔軟に変更可能にします。

## 3. テストと検証の戦略
### 3.1 ゴールデンマスター・テスト
リファクタリング前のプログラムの出力を「正解（Golden Master）」として保存し、変更後と比較します。
- **効率化**: フル実行（数年分のシミュレーション）は行わず、最初の 100 ステップ程度の内部状態および出力を保存します。
- **比較対象**: 全ての配列（`T`, `M`, `T_posi` 等）および `P_air` 等のスカラ値をバイナリまたは高精度な CSV でダンプします。

### 3.2 パラメータ空間の網羅的検証（網羅テスト）
惑星科学的にありうる範囲の入力に対して、リファクタリング後も一貫した結果（元の実装と同じ結果）を出すか確認します。
- **対象パラメータ**:
  - `P_total`: 0.001 - 10.0 [bar]
  - `obl` (傾斜角): 0 - 90 [deg]
  - `alpha` (斜面角度): 0 - 45 [deg]
- **検証手法**: 上記の格子点状の組み合わせに対して、短時間（例えば 1 火星日分）の計算を実行し、元実装との差異が許容誤差（浮動小数点誤差範囲）内であることを確認します。

## 4. 実施フェーズ
### フェーズ 1: 準備
- 現在の `main` 関数を整理し、内部状態をダンプする機能を追加。
- 基準となるテストデータを生成するスクリプトを作成。

### フェーズ 2: 構造体導入と関数シグネチャ変更
- `PlanetParams`, `ClimateState` を定義し、各関数がこれらを受け取るように変更。
- この時点でゴールデンマスター・テストをパスすることを確認。

### フェーズ 3: ソース分割（モジュール化）
- ソースファイルを物理過程ごとに分割。
- `Makefile` またはビルドスクリプトを作成。

### フェーズ 4: 網羅的検証の実施
- 広範なパラメータセットでの自動テストを実行。
- [COMPILATION_REPORT.md](../compilation_fix/COMPILATION_REPORT.md) で特定された警告（未使用変数等）の論理的削除。

## 5. 次の AI エージェントへの指示
1. まず、`test.c` をコンパイルし、特定のパラメータ（デフォルト設定）で 100 ステップ実行した際の状態を `check_origin.dat` に出力するツールを作成せよ。
2. 本計画の「2.1 状態と設定のカプセル化」に従い、構造体を定義せよ。
3. 全ての関数を構造体引数に移行し、1 のツールで生成した `check_origin.dat` との結果が 10^-12 以上の精度で一致することを確認せよ。
4. 網羅적パラメータ検証スクリプトを作成し、境界条件（極端な気圧や傾斜角）で数値的不安定性が発生しないか（元実装と同等か）を確認せよ。
